## 需求描述

为了支持全球化游戏运营，解决跨时区玩家在同一服务器活动中的时间不同步问题（如限时活动、竞技场刷新时间），需要构建一个全链路的实时时区调度系统。该系统需支持基于玩家所在时区动态调整 UI 显示时间，并在后端进行时区无关的业务逻辑计算。

## 技术方案

### 1. 动态时区适配中间件 (Backend Middleware)
- 实现一个 HTTP Middleware，自动检测用户请求头中的 `Time-Zone` 信息。
- 所有 API 返回的时间戳应采用 UTC 格式，由客户端根据用户时区进行本地化展示。
- 服务端核心业务逻辑（如任务倒计时）必须基于 UTC 毫秒级时间戳处理。

### 2. 时区感知型活动编排引擎 (Event Scheduler)
- 在 `game-event-service` 中引入时区策略配置，支持“绝对时间触发”和“用户本地相对时间触发”。
- 数据库表结构需在 `events` 表增加 `is_timezone_relative` 和 `target_timezone` 字段。
- 引入缓存策略，按时区预计算活动触发时间并缓存，降低实时计算开销。

## 验收标准

- [ ] 确保 API 接口统一返回 UTC 时间戳，不依赖服务端时区。
- [ ] 验证限时活动（如：北京时间 20:00 开启）在不同时区玩家客户端的准确显示。
- [ ] 系统时区配置支持热更新，无需重启服务。
- [ ] 完成跨时区业务场景的单元测试，覆盖 DST（夏令时）切换场景。

## 影响范围

- `user-service`: 获取/更新用户时区偏好
- `gateway`: 时区信息解析中间件注入
- `game-event-service`: 活动调度逻辑重构
