# REQ-00499：事件驱动服务编排与分布式状态机引擎

- **编号**：REQ-00499
- **类别**：可扩展性/解耦
- **优先级**：P1
- **状态**：new
- **涉及服务/模块**：backend/shared、catch-service、gym-service、social-service、reward-service、payment-service
- **创建时间**：2026-07-08 08:30
- **依赖需求**：REQ-00300（服务注册发现已实现）

## 1. 背景与问题

mineGo 项目中存在多个跨服务的复杂业务流程，例如：

1. **捕捉流程**：用户捕捉精灵 → 扣减道具（reward-service）→ 更新精灵库存（pokemon-service）→ 发送通知（social-service）→ 计算奖励（reward-service）
2. **Raid 战斗流程**：创建房间（gym-service）→ 匹配玩家（social-service）→ 开始战斗（gym-service）→ 结算奖励（reward-service）
3. **精灵交易流程**：发起交易（social-service）→ 双方确认 → 验证精灵归属（pokemon-service）→ 执行交换 → 发送通知 → 计算奖励
4. **支付流程**：创建订单（payment-service）→ 验证支付 → 扣减库存 → 发放道具 → 发送通知

当前这些流程的实现分散在各个服务中，存在以下问题：

- **紧耦合**：服务间通过 HTTP 直接调用，缺少统一的编排层，修改流程需要改动多个服务
- **状态管理分散**：流程状态分散在各服务数据库中，难以追踪完整流程生命周期
- **异常处理不完整**：部分流程失败后缺少统一的补偿/回滚机制
- **可观测性不足**：无法追踪跨服务的完整业务流程执行路径和耗时
- **版本管理困难**：业务规则变更时无法平滑升级流程版本

## 2. 目标

构建一个事件驱动的服务编排引擎，实现：

1. **流程定义框架**：支持 YAML/JSON 定义业务流程，包含步骤、状态转换、超时、补偿
2. **分布式状态机**：基于 Redis 持久化流程状态，支持跨服务状态同步
3. **事件驱动执行**：通过 EventBus 触发步骤执行，实现服务解耦
4. **补偿事务支持**：支持 Saga 模式的正向执行与逆向补偿
5. **流程可观测性**：完整的流程追踪、日志、指标、告警
6. **版本管理**：支持多版本流程共存，平滑升级

预期收益：
- 业务流程代码量减少 30%+
- 跨服务流程可追踪性提升 100%
- 流程异常恢复时间缩短 50%

## 3. 范围

- **包含**：
  - ProcessOrchestrator 编排引擎核心模块
  - DistributedStateMachine 状态机实现
  - 流程定义 DSL（YAML 格式）
  - 补偿事务管理器（Saga 模式）
  - 流程追踪与监控
  - 内置流程定义（捕捉、Raid、交易、支付）
  - 流程版本管理

- **不包含**：
  - BPMN 可视化流程设计器
  - 流程暂停/恢复的人工干预界面
  - 流程模拟与测试沙箱（后续需求）

### 4.3 流程定义 DSL

```yaml
# backend/shared/processes/catch-process.yaml
processType: catch-flow
version: 1.0.0
description: 精灵捕捉完整流程

states:
  - name: initiated
    transitions:
      - to: validating
        event: catch.initiated
  - name: validating
    timeout: 10s
    step:
      service: catch-service
      action: validateCatch
      input: [userId, pokemonId, location]
    transitions:
      - to: deducting-items
        event: catch.validated
      - to: failed
        event: catch.validation_failed
  - name: deducting-items
    timeout: 5s
    step:
      service: reward-service
      action: deductItems
      input: [userId, itemsUsed]
    compensation:
      service: reward-service
      action: refundItems
    transitions:
      - to: adding-pokemon
        event: items.deducted
  - name: adding-pokemon
    timeout: 15s
    step:
      service: pokemon-service
      action: addPokemon
      input: [userId, pokemonData]
    compensation:
      service: pokemon-service
      action: removePokemon
    transitions:
      - to: calculating-rewards
        event: pokemon.added
  - name: calculating-rewards
    step:
      service: reward-service
      action: calculateRewards
    transitions:
      - to: sending-notification
        event: rewards.calculated
  - name: sending-notification
    step:
      service: social-service
      action: sendNotification
    transitions:
      - to: completed
        event: notification.sent
  - name: completed
    final: true
  - name: failed
    final: true
    action: triggerCompensation
  - name: compensating
    transitions:
      - to: cancelled
        event: compensation.completed

timeout: 60s
retryPolicy:
  maxRetries: 3
  backoff: exponential
```

### 4.5 流程追踪

- 每个步骤执行记录：开始时间、结束时间、输入、输出、错误
- 流程实例追踪 ID：关联所有相关日志
- Prometheus 指标：
  - `process_started_total{type}`
  - `process_completed_total{type,status}`
  - `process_step_duration_ms{type,step}`
  - `process_compensation_total{type}`
- 链路追踪集成：每个流程实例生成 traceId

### 4.6 数据库迁移

```sql
-- database/migrations/XXX_create_process_tables.sql
CREATE TABLE process_instances (
  instance_id VARCHAR(64) PRIMARY KEY,
  process_type VARCHAR(64) NOT NULL,
  version VARCHAR(32) NOT NULL,
  status VARCHAR(32) NOT NULL,

  step_name VARCHAR(64) NOT NULL,

  action VARCHAR(64),
  input JSONB,
  output JSONB,
  error JSONB,

  completed_at TIMESTAMP,
  compensation_status VARCHAR(32),
  compensation_completed_at TIMESTAMP,

CREATE INDEX idx_process_instances_type ON process_instances(process_type);
```
