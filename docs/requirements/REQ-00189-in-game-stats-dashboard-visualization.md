# REQ-00189：游戏内实时数据可视化与统计仪表板

- **编号**：REQ-00189
- **类别**：前端体验
- **优先级**：P1
- **状态**：new
- **涉及服务/模块**：game-client、frontend/game-client/src/components/StatsDashboard.js、frontend/game-client/src/charts、gateway、user-service、pokemon-service、reward-service
- **创建时间**：2026-06-14 09:05
- **依赖需求**：无

## 1. 背景与问题

当前游戏客户端缺少一个综合性的数据可视化与统计仪表板，玩家无法直观地查看自己的游戏进度、成就统计、精灵收集进度等关键数据。虽然后端已有丰富的数据（精灵图鉴、成就系统、排行榜等），但前端展示分散在各个页面，缺乏统一的数据洞察视图。

从代码分析来看：
- `pokemon-service` 已实现 `pokedexService.js` 和 `achievementService.js`
- `reward-service` 有排行榜和任务系统
- `game-client` 的 `main.js` 仅展示了基础资源（精灵球、金币、星尘）
- 缺少图形化的数据展示组件（图表、进度条、趋势图等）

玩家需要：
1. 一眼看到自己的游戏进度和核心指标
2. 查看精灵收集完成度、成就进度
3. 追踪近期的游戏活动（捕捉数、道馆战绩等）
4. 对比自己与好友/全服的数据

## 2. 目标

构建一个功能完善的实时数据可视化仪表板系统，提供：
- 游戏进度概览（等级、经验、收集进度）
- 精灵图鉴完成度可视化（类型分布、稀有度统计）
- 成就系统进度展示
- 近期活动数据图表（捕捉趋势、道馆战绩）
- 个人与全服数据对比

## 3. 范围

- **包含**：
  - 前端 StatsDashboard 组件开发
  - 图表库集成（Chart.js 或轻量级替代方案）
  - 数据聚合 API 端点设计
  - 响应式布局适配移动端
  - 实时数据更新机制

- **不包含**：
  - 管理后台的数据仪表板（属于 admin-dashboard）
  - 详细的战斗数据分析（属于单独的战斗统计系统）
  - 导出报告功能（后续需求）

## 4. 详细需求

### 4.1 前端组件

#### 4.1.1 StatsDashboard 主组件
- 入口按钮：在主界面顶部栏添加"统计"图标
- 布局：可滚动卡片式设计，每个卡片展示一类数据
- 支持下拉刷新和自动定时刷新（5分钟间隔）

#### 4.1.2 核心数据卡片
1. **游戏进度卡片**
   - 当前等级和经验进度条
   - 总游戏时长
   - 本周活跃天数

2. **精灵收集卡片**
   - 图鉴完成度环形图（已发现/总数）
   - 按类型分布的柱状图
   - 稀有精灵数量统计

3. **成就进度卡片**
   - 成就完成度百分比
   - 近期获得的成就列表（带时间戳）
   - 推荐下一个可完成的成就

4. **活动统计卡片**
   - 近7天捕捉数量趋势折线图
   - 道馆战绩胜率饼图
   - 本月累计获得的奖励

5. **社交对比卡片**
   - 好友排名对比柱状图
   - 全服排名位置指示器

#### 4.1.3 图表技术要求
- 使用 Chart.js CDN 或打包（gzip < 30KB）
- 支持触摸手势（缩放、滑动）
- 图表动画时长 < 300ms
- 离线时使用缓存数据，标记"上次更新时间"

### 4.2 后端 API

#### 4.2.1 数据聚合端点
```
GET /api/v1/user/stats/dashboard
```
响应结构：
```json
{
  "progress": {
    "level": 25,
    "expCurrent": 45000,
    "expNext": 50000,
    "totalPlayTime": "120h 30m",
    "activeDaysThisWeek": 5
  },
  "collection": {
    "discovered": 89,
    "total": 151,
    "byType": { "fire": 12, "water": 15, ... },
    "rarity": { "common": 40, "rare": 30, "legendary": 3 }
  },
  "achievements": {
    "completed": 45,
    "total": 100,
    "recent": [...],
    "recommended": [...]
  },
  "activity": {
    "weeklyCatches": [12, 15, 8, 20, 18, 10, 5],
    "gymStats": { "wins": 50, "losses": 20, "draws": 5 },
    "monthlyRewards": { "coins": 500, "stardust": 10000 }
  },
  "ranking": {
    "friends": [...],
    "serverPosition": 1250
  }

```

#### 4.2.2 数据缓存策略
- Redis 缓存：TTL 5分钟
- 缓存键：`stats:dashboard:{userId}`
- 用户请求时检查缓存，过期则重新聚合

### 4.3 性能要求
- 首屏加载时间 < 1.5秒
- 图表渲染时间 < 500ms
- API 响应时间 < 300ms（缓存命中）/< 800ms（缓存未命中）
- 内存占用 < 20MB（图表组件）

### 4.4 可访问性
- 图表提供数据表格替代视图
- 支持高对比度模式
- 屏幕阅读器可读取关键数据

## 7. 优先级理由

**P1（高优先级）**

理由：
1. **用户体验显著提升**：数据可视化是现代游戏的标配，缺少此功能影响用户留存
2. **数据价值释放**：后端已有丰富数据，前端可视化能最大化数据价值
3. **核心功能补充**：与图鉴、成就、排行榜等已完成系统形成闭环
4. **用户反馈需求**：玩家常希望直观看到自己的游戏进度
5. **成熟度贡献**：提升"前端体验"维度评分，助力项目达到生产可用标准
