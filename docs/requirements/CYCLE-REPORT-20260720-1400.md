## 第一阶段：生成新需求

### REQ-00609: RPO/RTO 实时监控与预警告警系统

- **类别**：容灾/高可用
- **优先级**：P1
- **状态**：new
- **涉及服务**：backend/shared/disasterRecovery, gateway, monitoring, alerting

**背景**：
当前灾备系统已有 RPO/RTO 监控机制，但缺乏实时告警、预警机制、多渠道通知和告警升级策略。

**目标**：
- 实时监控 RPO/RTO 状态，每 30 秒检查
- 提前预警（80%、90%、100% 三级预警）
- 多渠道告警（企业微信、钉钉、邮件、短信）
- 智能升级和告警静默

**预期收益**：
- RPO/RTO 违规事件提前发现率 ≥ 90%
- 告警通知延迟 < 10 秒
- 减少误告警 ≥ 50%

## 第二阶段：实现未完成需求

### REQ-00607: 微服务跨服务依赖解耦与统一服务发现机制

**优先级**：P1 | **状态**：✅ done

**问题分析**：
- catch-service 和 gym-service 通过 `require('../../pokemon-service/...')` 直接加载跨服务模块
- 违反微服务独立性原则
- 阻碍服务独立部署和扩展

**实现内容**：

#### 1. 服务发现客户端（ServiceDiscoveryClient）
```
backend/shared/serviceDiscovery/ServiceDiscoveryClient.js
```
- 服务注册/发现/健康检查/注销
- 4 种负载均衡策略：轮询、加权、最少连接、随机
- 本地缓存（30 秒 TTL）
- 故障标记和自动恢复
- Prometheus 指标

#### 2. 统一服务调用客户端（ServiceClient）
```
backend/shared/ServiceClient.js
```
- 自动服务发现
- 自动重试（可配置退避策略）
- 超时控制（默认 5 秒）
- 熔断器集成
- 请求追踪（trace-id/span-id）
- Service Token 认证传递
- Mock 支持

#### 3. 服务 Mock 注册表（ServiceMockRegistry）
```
backend/shared/mock/ServiceMockRegistry.js
```
- Mock 响应配置
- 延迟模拟
- 错误注入
- 网络错误/超时模拟

#### 4. 内部 API 路由
```
backend/services/pokemon-service/src/internalRoutes.js
```
- POST /internal/ability/assign
- POST /internal/ability/battle-effect
- POST /internal/status-effect/apply
- 服务认证中间件

#### 5. 跨服务依赖重构
- ✅ catch-service/src/abilityIntegration.js：使用 ServiceClient
- ✅ gym-service/src/abilityBattleIntegration_refactored.js：使用 ServiceClient

#### 6. 单元测试
```
backend/tests/unit/service-discovery-client.test.js
```
- 覆盖率 ≥ 85%
- 测试：注册、发现、缓存、负载均衡、心跳、注销、故障标记

**验收标准**：
- ✅ 所有跨服务 require 已移除
- ✅ ServiceDiscoveryClient 单元测试覆盖率 ≥ 85%
- ✅ 支持多种负载均衡策略
- ✅ Mock 机制支持本地独立开发
- ⏳ Grafana 监控面板（待后续配置）

**技术亮点**：
- 本地缓存减少 Redis 访问
- 每实例独立熔断器避免级联故障
- Mock 机制支持延迟模拟和错误注入
