# REQ-00556：基于机器学习的实时行为异常检测系统

## 元信息

| 字段 | 值 |
|------|-----|
| 编号 | REQ-00556 |

| 类别 | 反作弊 |

| 状态 | new |

## 1. 背景与问题

当前 mineGo 项目已实现多种反作弊检测机制：
- GPS 伪造检测与速度限制 (`anti-cheat.js`)
- 精灵交换欺诈检测 (`tradeFraudDetection.js`)
- 协同作弊团伙检测 (`gangDetection/`)
- 设备完整性验证 (`deviceIntegrity.js`)

然而，这些系统主要依赖**规则引擎**和**阈值判定**，存在以下问题：

1. **滞后性**：新型作弊手段无法被规则覆盖，需要人工更新规则
2. **误报率**：静态阈值难以适应不同用户的正常行为差异
3. **单维度**：现有系统关注单一行为特征，缺乏综合行为画像
4. **被动防御**：检测逻辑固定，作弊者可通过测试找到规避方法

## 2. 目标

构建基于机器学习的实时行为异常检测系统，实现：

1. **自适应学习**：根据用户群体行为自动调整检测基线
2. **多维特征融合**：综合分析位置、时间、设备、社交、游戏行为等多维数据
3. **实时检测**：在用户行为发生时即时评估异常风险
4. **零日检测**：识别未知新型作弊模式
5. **低误报**：通过用户行为画像降低对正常用户的误判

## 3. 范围

### 包含
- 用户行为特征提取与画像系统
- 实时异常检测推理引擎
- 模型训练与在线学习管道
- 异常事件标注与反馈闭环
- 风险评分与分级响应机制
- Prometheus 指标与监控告警

### 不包含
- 前端管理界面（作为后续需求）
- 离线批量分析任务（作为后续需求）
- 图像识别类作弊检测（如自动捕捉）
- 与第三方风控服务的集成

## 4. 详细需求

### 4.1 行为特征提取模块

提取以下维度特征（每用户实时更新）：

**位置特征**：
- 移动速度分布（均值、方差、峰值）
- 位置熵（位置不确定性）
- 跳跃距离序列
- 在线时长与移动距离比

**时间特征**：
- 活跃时段分布
- 行为间隔时间统计
- 异常活跃时段检测（凌晨、节假日）

**设备特征**：
- 设备指纹变化频率
- 设备型号与 OS 版本异常
- 设备传感器数据一致性

**游戏行为特征**：
- 捕捉成功率趋势
- 精灵稀有度与捕捉能力匹配度
- 道具使用模式
- 道馆战斗表现

**社交特征**：
- 好友网络拓扑
- 交互频率分布
- 协同行为模式

### 4.2 异常检测推理引擎

采用**在线学习**架构：
- 输入：用户实时行为事件流
- 处理：增量更新用户画像 + 实时推理
- 输出：异常风险分数 (0-100)

算法选择：
- **Isolation Forest**：用于异常点检测
- **One-Class SVM**：用于正常行为边界学习
- **Autoencoder**：用于行为序列重构误差分析
- **在线聚类**：用于行为模式分组

推理延迟要求：
- P95 < 100ms
- P99 < 500ms

### 4.3 模型训练管道

训练流程：
1. 数据采集：从 PostgreSQL 和 Kafka 收集行为数据
2. 特征工程：计算上述特征维度
3. 标注数据：结合已有反作弊规则结果 + 人工标注
4. 模型训练：每日增量训练，每周全量训练
5. 模型验证：在测试集上验证准确率、召回率、误报率
6. 模型发布：通过 ConfigCenter 热更新

模型版本管理：
- 使用 MLflow 或自建版本管理
- 支持 A/B 测试和灰度发布
- 模型回滚机制

### 4.4 风险评分与分级响应

风险等级定义：

| 等级 | 分数范围 | 响应措施 |
|------|----------|----------|
| 正常 | 0-30 | 正常通过 |
| 低风险 | 31-50 | 记录日志，不限制 |
| 中风险 | 51-70 | 功能限制（如捕捉冷却延长） |
| 高风险 | 71-85 | 临时封禁，触发人工审核 |
| 极高风险 | 86-100 | 永久封禁，记录证据 |

响应动作：
- 实时阻断可疑请求
- 触发 CAPTCHA 验证
- 降低捕捉成功率
- 限制道馆战斗
- 发送警告通知

### 4.5 反馈闭环机制

- 自动标注：结合现有反作弊规则结果
- 人工标注：管理员审核系统，标注真实作弊样本
- 用户申诉：用户可申诉被封禁，申诉结果反馈到训练数据
- 假阳性学习：正常用户被误判后，自动加入负样本

### 4.6 监控与可观测性

Prometheus 指标：
- `ml_anomaly_detection_requests_total`：检测请求总数
- `ml_anomaly_detection_latency_seconds`：推理延迟分布
- `ml_anomaly_risk_score_histogram`：风险分数分布
- `ml_anomaly_blocked_total`：被阻止的请求总数
- `ml_model_version`：当前模型版本
- `ml_model_accuracy`：模型准确率
- `ml_model_false_positive_rate`：误报率

告警规则：
- 推理延迟 P95 > 200ms
- 误报率 > 5%
- 模型准确率 < 80%

## 5. 验收标准

- [ ] 能够在 100ms 内完成实时异常检测推理（P95）
- [ ] 对已知作弊类型的检测准确率 >= 95%
- [ ] 对正常用户的误报率 <= 2%
- [ ] 支持至少 5 种行为特征的实时提取
- [ ] 模型支持热更新，更新过程不中断服务
- [ ] 提供 Prometheus 指标和 Grafana 看板
- [ ] 通过负载测试：1000 QPS 不降级

## 6. 工作量估算

**L (Large)**

理由：
- 涉及机器学习基础设施搭建
- 需要实现在线学习管道
- 多模块协同（特征、推理、训练、反馈）
- 预计开发时间：2-3 周

## 7. 优先级理由

**P1（高优先级）**

1. 反作弊是游戏核心安全需求，直接影响游戏公平性和收入
2. 现有规则系统已覆盖基础场景，需要升级以应对更复杂的作弊手段
3. 与已完成的 REQ-00550 协同作弊检测形成互补，构建完整风控体系
4. 实施后可显著降低人工审核成本，提升作弊检测效率
