# REQ-00033：API 压力测试与性能基准系统

- **编号**：REQ-00033
- **类别**：测试覆盖
- **优先级**：P2
- **状态**：new
- **涉及服务/模块**：backend/tests/performance、所有微服务、GitHub Actions、docs/performance
- **创建时间**：2026-06-07 20:10
- **依赖需求**：REQ-00002（结构化日志与 Prometheus 指标）、REQ-00022（集成测试框架）

## 1. 背景与问题

当前项目已有完善的单元测试（145 个）和集成测试（42 个），但缺少系统级的压力测试和性能基准：

1. **性能瓶颈未知**：无法确定各 API 在高并发下的表现，延迟分布、吞吐量上限未量化
2. **容量规划缺失**：无法回答"系统能支持多少并发用户"的问题
3. **回归检测不足**：性能劣化无法在 CI 中自动检测
4. **SLA 定义空白**：没有明确的性能 SLA（如 P99 延迟 < 200ms）

根据 STATUS.md 的"未覆盖高价值缺口"，API 压力测试是测试覆盖维度的关键缺失项。

## 2. 目标

建立完整的 API 压力测试体系：
- 使用 k6 或 Artillery 构建可重复的压力测试
- 覆盖所有核心 API 端点（捕捉、道馆、支付、用户认证等）
- 建立 CI 集成的性能基准检测
- 定义并验证性能 SLA（吞吐量、延迟、错误率）
- 生成可视化的性能报告

## 3. 范围

### 包含
- k6 压力测试脚本编写（核心 API）
- 性能基准配置文件（吞吐量、延迟阈值）
- GitHub Actions CI 集成（每日/每周性能回归检测）
- Grafana 性能测试仪表板
- 性能测试文档和 SLA 定义

### 不包含
- 前端 E2E 性能测试（Playwright/Cypress）
- 真实生产环境压力测试
- 数据库专项性能测试（已有 REQ-00020 索引优化）

## 4. 详细需求

### 4.1 压力测试脚本

使用 k6 编写压力测试脚本，覆盖以下核心场景：

```javascript
// 压力测试场景示例
// 1. 用户认证流程（注册/登录）
// 2. 精灵捕捉流程（位置上报 → 发现精灵 → 投球捕捉）
// 3. 道馆战斗流程（进入道馆 → 挑战 → 占领）
// 4. 支付流程（创建订单 → 支付 → 回调）
// 5. 社交操作（添加好友 → 发送礼物）
// 6. 奖励领取（每日任务 → 成就领取）
```

测试类型：
- **负载测试**：模拟正常负载（100-500 并发用户）
- **压力测试**：逐步增加负载直到系统崩溃
- **峰值测试**：短时间内突发高流量
- **浸泡测试**：长时间持续负载（检测内存泄漏）

### 4.2 性能基准配置

定义各 API 的性能 SLA：

| API | 吞吐量目标 | P50 延迟 | P99 延迟 | 错误率 |
|-----|-----------|---------|---------|--------|
| POST /auth/login | 200 req/s | < 50ms | < 150ms | < 0.1% |
| POST /catch/attempt | 150 req/s | < 100ms | < 300ms | < 0.5% |
| GET /pokemon/nearby | 300 req/s | < 80ms | < 200ms | < 0.1% |
| POST /gym/battle | 100 req/s | < 150ms | < 400ms | < 0.5% |
| POST /payment/create | 50 req/s | < 200ms | < 500ms | < 0.1% |
| GET /social/friends | 200 req/s | < 50ms | < 150ms | < 0.1% |

### 4.3 CI/CD 集成

GitHub Actions 工作流：
```yaml
name: Performance Tests
on:
  schedule:
    - cron: '0 2 * * *'  # 每天凌晨 2 点
  workflow_dispatch:
  pull_request:
    paths:
      - 'backend/services/**'
      - 'backend/gateway/**'
```

性能回归检测：
- 对比最近 7 天的性能数据
- P99 延迟增长超过 20% → CI 失败
- 吞吐量下降超过 15% → CI 失败

### 4.4 性能监控仪表板

Grafana 仪表板展示：
- 实时吞吐量曲线
- 延迟分布（P50/P90/P95/P99）
- 错误率趋势
- 系统资源使用（CPU/内存/网络）
- 性能对比（当前 vs 基准）

### 4.5 报告生成

每次测试完成后生成：
- HTML 性能报告
- JSON 性能数据（用于趋势分析）
- 性能劣化告警通知

## 6. 工作量估算

理由：

- 需要搭建测试环境（独立 DB/Redis 实例）
- 需要集成 CI/CD 和监控

预估工时：2-3 天

## 7. 优先级理由

**P2**（非紧急但重要）：

1. 项目已达生产可用标准，性能测试是优化而非核心功能
2. 性能问题可在生产监控中逐步发现，不影响上线
3. 但对长期维护和质量保障非常重要
4. 作为测试覆盖维度的补强，提升项目成熟度

对"项目可用"的贡献：完善测试覆盖，从 11 分提升至 12-13 分。
