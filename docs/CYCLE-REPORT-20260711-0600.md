### 背景与问题

当前 mineGo 项目的 API 响应返回完整数据对象，存在以下问题：

1. **过度获取（Over-fetching）**：用户列表页只需基本信息，但返回完整数据（~5KB），移动端网络传输浪费 ~80%
2. **列表接口性能问题**：30 条精灵数据响应 150KB+，移动端加载慢
3. **不同场景需求差异**：列表页、详情页、战斗页、社交页需要不同字段
4. **缺少 GraphQL 级灵活性**：项目采用 REST API，无法像 GraphQL 按需查询字段

### 解决方案

构建 API 响应字段投影系统：

- **按需返回字段**：客户端通过 `fields` 参数指定需要的字段
- **预定义字段集**：支持场景化字段集（list/detail/battle/social）
- **嵌套字段投影**：支持关联对象的字段选择（如 `stats.hp`）
- **自动裁剪**：中间件自动裁剪响应数据
- **数据库查询优化**：Sequelize attributes 支持只查询需要的字段
